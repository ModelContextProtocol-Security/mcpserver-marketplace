This is a **Cloud Security Alliance (CSA) MCP Security Working Group** project evaluating the security of Model Context Protocol (MCP) marketplaces. It's also a demonstration of AI-driven project execution.

### The Core Problem

The MCP ecosystem exploded from **0 to 16,000+ servers in less than one year**, but with minimal security infrastructure. The central question is: **"How do I ultimately safely get an MCP server? How do I make sense of this?"**

## What Has Been Accomplished (Phase 1 - Foundation)

### Major Documents Created:

1. **GOALS.md** - Strategic direction with 5 primary goals:
   - Evaluate and rank MCP marketplaces
   - Identify security gaps
   - Create guidance for marketplace operators
   - Create guidance for users
   - Build official vendor MCP server reference

2. **EVALUATION-CRITERIA.md** - Detailed criteria across 6 categories plus **8 critical security gaps identified**:
   - Missing hosted servers in marketplaces (Critical)
   - Logo misuse creating false trust signals (High)
   - Provenance information regression (High)
   - Lack of reporting mechanisms (Medium)
   - No vendor verification process (High)
   - Domain validation complexity (Medium)
   - No official/community distinction (High)
   - No shared verification registry (Medium)

3. **Massive Security Research** - 70,000+ word analysis of PyPI/npm security (2015-2025):

   - Documented what security measures worked vs. failed
   - Applied lessons to MCP context

4. **MCP-MARKETPLACE-TAXONOMY.md** - Broad definition of "marketplace":
   - **8 marketplace types** (Registry, Built-in, Catalog, Code Hosting, AI Agents, Search/Discovery, Communities, Tutorials)
   - Key insight: In an AI-driven ecosystem, ANY resource that influences MCP server selection functions as a marketplace

5. **PROJECT-SPECIFICATION.md** - Technical requirements for **8 deliverables**:

   - Marketplace operator guidance
   - User guidance documents

   - Automation tools

6. **PROJECT-PLAN.md** - AI-driven execution roadmap with **7 phases**, no artificial timelines

## Key Insights Discovered

### The "Wild West" Problem
All major MCP marketplaces reviewed have a **critical gap**: they systematically ignore vendor-hosted servers and only list GitHub repositories for local installation. This drives users toward third-party implementations instead of official, vendor-hosted options.

### Package Registry Lessons
- **Optional security features fail** (only 10-15% adoption)
- **Mandatory graduated rollout works** (npm 2FA success)
- **Community-only detection doesn't scale**
- **Trusted Publishing eliminates credential theft**
- **Security requires dedicated, funded roles**

### AI Distribution Challenge
Unlike traditional package managers, **AI agents can autonomously install MCP servers** based on any information they find online, blurring the distinction between "informational content" and "marketplace."

- **Phase 7**: Ongoing monitoring and maintenance

The project has completed substantial foundational research and is ready to move into methodology development and actual marketplace evaluation.

### From EVALUATION-CRITERIA.md:

**The Problem:**

**Specific Examples Documented:**

1. **Airtable MCP server in Claude Desktop**:
   - Uses the Airtable logo prominently

   - No clear indication it's third-party, not official Airtable

2. **PubMed connector**:
   - Uses the Anthropic logo instead of NIH/PubMed logo
   - Misleads users about who created it

**The User Impact Quote:**
> "If I'm some junior user who's not looking into that level of detail, I look at that and think that's legit"

### Evaluation Criteria (Section 1.2)

The project includes **Logo Usage and Branding** as a specific evaluation criterion:

**Scoring Approach:**
- ✅ **Excellent**: Clear policy on logo usage; only official servers use vendor logos OR clear "unofficial" labeling
- ⚠️ **Partial**: Logos used but with some disclaimer or attribution
- ❌ **Poor**: Vendor logos used without verification or distinction between official/unofficial

### Recommendations Developed

**For Marketplace Operators:**
- Require vendor authorization before allowing logo usage
- Implement clear "unofficial" labeling for third-party servers
- Establish logo usage policies

**The Broader Issue:**
> "For all we know, maybe Airtable did bless this guy's MCP server, but we have no way to figure it out"

There's no mechanism to verify whether a vendor has sanctioned the use of their branding.

## Why This Matters

**Risk to Users:**
- Junior/non-technical users cannot distinguish official from unofficial
- Users make uninformed trust decisions based on visual branding

**Risk to Vendors:**
- Vendor reputation at stake from poorly-maintained third-party implementations
- No control over who uses their branding
- Potential liability issues

**Risk to Ecosystem:**
- Erodes trust in MCP ecosystem overall
- Makes it harder to identify actually malicious servers (everything looks "official")

This is actually one of the **critical gaps** the project identified and plans to address in both the evaluation criteria and the guidance documents for marketplace operators.
